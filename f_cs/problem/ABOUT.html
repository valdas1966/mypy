What is a CS Problem?

Every algorithm starts with a question. The Problem is that question — formalized.

1. The Idea

🧾 Think of a restaurant order. The Problem is the order slip — it tells the kitchen what to cook (inputs, constraints, goal), but not how to cook it. The Algorithm is the chef. The Solution is the plate that comes out.
📋
Problem
What to solve.
Inputs, constraints, goal.
⚙️
Algorithm
How to solve it.
Step-by-step procedure.
✅
Solution
The answer.
Valid or not, with stats.

The 3 Parts of Any CS Problem

  1. Input — the data you start with (a graph, a list, a grid, a number).
  2. Constraints — the rules (edges have weights, items have sizes, cells are blocked).
  3. Objective — what "solved" means (shortest path, minimum cost, valid assignment).

2. The Flow: Problem → Algorithm → Solution

solution.problem      # the original problem
solution.stats.elapsed  # how long it took

3. This Module: ProblemAlgo

ProblemAlgo — the abstract root

  1. 29 lines of code. Intentionally minimal.
  2. Defines the contract: every problem has a name and a key for identity.
  3. Never instantiated directly — always subclassed with domain-specific details.
  4. Domain-agnostic: works for pathfinding, scheduling, constraint satisfaction, anything.
class ProblemAlgo(HasName, Equatable):

        raise NotImplementedError
HasName → name, __str__  |  Equatable → __eq__, __hash__ via key

How Subclasses Extend It

ProblemAlgo ── name + equality └── ProblemSearch ── adds: grid, successors [f_search] ├── ProblemSPP ── adds: start, goal [shortest path] │ └── ProblemSPPLookup ── adds: cached distances └── ProblemOMSPP ── adds: start, goals[] [one-to-many]
  1. ProblemAlgo says: "I have a name and can be compared."
  2. ProblemSearch says: "I also have a grid to search on."
  3. ProblemSPP says: "I also have a start and a goal."
  4. Each layer adds exactly what it needs — nothing more.

4. Hall of Fame: Legendary CS Problems

🗺️
Shortest Path Problem
Dijkstra, 1956
Find the cheapest route between two nodes in a graph. GPS navigation, network routing, and game AI all depend on it. Dijkstra solved it while drinking coffee at a cafe in Amsterdam — on paper, in 20 minutes.
O(E log V) — solved
🧳
Travelling Salesman (TSP)
Formalized 1930s
Visit every city exactly once and return home, minimizing total distance. Sounds simple — but with 20 cities there are 60 quadrillion possible routes. The poster child of NP-hard problems.
NP-hard
👸
N-Queens Problem
Max Bezzel, 1848
Place N queens on an N×N chessboard so none threaten each other. Originally a chess puzzle from 1848 — became a benchmark for backtracking algorithms. For N=8, there are 92 solutions.
backtracking — solved
🎒
Knapsack Problem
Tobias Dantzig, 1897
Pack the most value into a bag with a weight limit. The 0/1 version is NP-complete; the fractional version is greedy-solvable. Used in resource allocation, finance, and cargo loading.
NP-complete (0/1)
⏱️
P vs NP
Cook & Levin, 1971
Can every problem whose solution is quick to verify also be quick to solve? The biggest open question in CS. A $1,000,000 Millennium Prize awaits whoever cracks it. Most experts believe P ≠ NP.
$1M prize — open
🛑
Halting Problem
Alan Turing, 1936
Can a program decide if another program will halt or loop forever? Turing proved it's impossible — the first undecidability result. This proof also gave us the concept of the Turing Machine itself.
undecidable
🔀
Sorting
~1945 onwards
Arrange items in order. Seems trivial, but it drove decades of algorithm design: Quicksort, Mergesort, Heapsort, Timsort. The proven lower bound is O(n log n) for comparison-based sorts.
Ω(n log n) — solved
🌉
Seven Bridges of Königsberg
Euler, 1736
Cross all 7 bridges exactly once and return. Euler proved it's impossible — and accidentally invented graph theory. This is the problem that started it all: the very first CS problem, 200 years before computers.
impossible — proved 1736

5. Fun Facts

☕
Dijkstra's 20-Minute Algorithm
Edsger Dijkstra invented his shortest-path algorithm in 1956 while sitting at a cafe with his fiancée in Amsterdam. He designed it in 20 minutes on paper — no computer. He didn't publish it until 1959 because he thought it was "too simple" for a paper.
🐜
Ants Solve TSP
Ant Colony Optimization (1992) solves the Travelling Salesman Problem using virtual ants that leave pheromone trails. Shorter paths get more pheromones, attracting more ants. Real ants actually do this — computer scientists copied them.
🃏
Sorting a Deck: Bogosort
The worst sorting algorithm: shuffle randomly, check if sorted, repeat. Expected time: O(n × n!). For 52 cards, you'd need about 1068 shuffles on average. The universe will end first. It exists purely as a joke — and an exam question.
💰
Million-Dollar Problems
The Clay Mathematics Institute offers $1,000,000 each for 7 Millennium Prize Problems. P vs NP is one. So far only one has been solved: the Poincaré Conjecture (Grigori Perelman, 2003). Perelman declined the money.
📦
NP-Hard in Real Life
Every time a delivery company routes its trucks, it's solving a variant of TSP. UPS saved $400 million per year by using heuristic algorithms that produce "good enough" solutions to NP-hard routing problems. Perfection isn't required — just close.
🧬
DNA is a CS Problem
Genome assembly (piecing together DNA from fragments) is equivalent to finding an Eulerian path in a de Bruijn graph — connecting directly back to Euler's bridge problem from 1736. Biology and CS are running the same algorithms.
