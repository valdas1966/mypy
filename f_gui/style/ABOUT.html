The reuse story — one Stroke, many users

The whole package exists so the (color, width, style) triple is defined exactly once.

🖊️

A Stroke is a pen-tip setting. A Line is that pen plus two endpoints and an arrow. A Border is just four of those pens, one per edge. Define the triple once; lines and borders both borrow it — no duplicated color/width/style.

Where does style belong?

A natural question: should this sit under elements/, since elements use it? The dependency graph answers it.

Verdict: keep f_gui/style/ as a peer of elements/ and render/ — do not nest it under elements. Style is a dependency of elements, not a part of them, and it is already used outside the element context.
f_gui/ ├── style/ ← value objects (Stroke, LineStyle, Border, TextStyle) ├── elements/ ── imports ──▶ style └── render/ ── imports ──▶ style f_quiz/ ── imports ──▶ style (a different top-level module!)
  1. One-way dependency. elements→style, render→style, f_quiz→style. Style imports nothing back — nesting it inside a consumer inverts the layering.
  2. Used outside elements. The renderer (f_gui/render/html) and a different top-level module (f_quiz/question_diagram) both consume style without any Element.
  3. Repo precedent. The other primitives elements depend on — f_ds.geometry (Bounds/Point/Side) — live in their own layer, not under elements.
  4. Future caveat. If a non-GUI consumer ever appears, promote it to a top-level f_style. Until then that's YAGNI — f_gui/style is the right altitude.

Why it's built this way

♻️

Value objects

Small immutable bundles of data, shared by reference — define the (color,width,style) triple once, reuse it everywhere.

🎯

Zero-translation enums

LineStyle stores the literal CSS keywords solid/dashed/dotted, so the renderer maps them with no lookup.

✂️

Data ⟂ rendering

Style stores appearance; the stateless RenderHtml turns it into CSS/SVG and imports no style type — it duck-types the values.

🧱

Composition

A Border isn't a new primitive — it's literally four Strokes grouped, one per edge.

🎨

Content ⟂ presentation

TextStyle is the CSS idea (since 1996): keep the words on the Label, keep the look in a separate, reusable object.

🕳️

The sentinel trick

TextStyle's color has three states — omitted→black, None→inherit, RGB→that — so it needs a sentinel, since Python can't tell "not passed" from "passed None".
