What is a Connector?

A PowerPoint-style smart arrow between two elements — it follows them when they move.

●  connector · #fbbf24 leaf Element auto-routed

The Idea

An arrow that knows what it points at, not just where it is.

🧲

A Connector is a magnetic cable. Plug one end into box A and the other into box B, then drag the boxes anywhere on the slide — the cable stays clipped to both and re-routes itself. You never re-draw it; it always knows where its two ends live.

❌ A fixed-coordinate Line

  • Stores two frozen Points.
  • Move a box and the line stays put — it detaches and dangles.
  • You must hunt down and rewrite both endpoints by hand.
  • The drawing and the layout drift out of sync.

✅ A Connector (references, not points)

  • Holds references to the two Elements.
  • Pulls fresh anchor points from each element's bounds on every access.
  • path recomputes itself — so it tracks movement automatically.
  • No event system, no observers, no manual sync.

The rules it lives by

  1. It references two Elements (src, dst) — not two fixed points.
  2. Endpoints are pulled live from each element's bounds, so path recomputes on each access and tracks the elements as they move.
  3. Each attach Side may be given explicitly or left None to auto-pick the nearest facing sides: a horizontal gap → RIGHT/LEFT, a vertical gap → BOTTOM/TOP.
  4. Routing.DIRECT draws one straight segment; Routing.ORTHOGONAL draws a 90° elbow.
  5. src, dst and the Connector must share a parent — the same 0–100 coordinate frame.
  6. bounds is computed (it overrides Element.bounds), so it stays in sync with the live path.

This Module's Contribution

The Routing enum and a path that reads its endpoints live.

i_1_connector Routing live path 

class Routing(Enum):
    """Routing — how a Connector shapes its path between its two anchors."""
    DIRECT     = 'direct'      # a single straight segment (anchor -> anchor)
    ORTHOGONAL = 'orthogonal'  # a 90-degree elbow (1-2 bends)


@property
def path(self) -> list[Point]:
    """Polyline vertices (>= 2), computed live from the elements' bounds."""
    side_src, side_dst = self._sides()
    p_src = self._src.bounds.anchor(side=side_src)   # read LIVE
    p_dst = self._dst.bounds.anchor(side=side_dst)   # read LIVE
    if self._routing is Routing.ORTHOGONAL:
        return Connector._orthogonal(p_src=p_src, side_src=side_src,
                                     p_dst=p_dst, side_dst=side_dst)
    return [p_src, p_dst]                       # DIRECT

The ORTHOGONAL elbow, step by step

  1. Each anchor steps out by _STUB (6 units) along its side's outward normal — so the line leaves the box perpendicular to the side it's attached to.
  2. The two stubs are joined: a vertical mid-line for two horizontal normals, a horizontal mid-line for two vertical normals, or a single corner for a mixed pair.
  3. Collinear points are dropped (_simplify) so the polyline carries only its real bends.

Obstacle avoidance is out of scope — the elbow does not try to dodge the boxes or anything between them.

Fun Facts

A few things worth knowing about smart connectors.

📏

"Manhattan" routing

Right-angle (orthogonal) paths are the standard in diagram editors and in chip layout (EDA) — like walking a grid of city blocks.

♻️

No observers needed

The recompute-on-access trick means there are no events or listeners: the path is derived fresh on every single read.

🧗

Routing is genuinely hard

General obstacle-avoiding routing is a pathfinding problem — which is exactly why it's left out of scope here.

📍

Perpendicular exits

A connector leaves a box at a right angle to the side it attaches to — that's the _STUB stepping along the side's normal.

🧮

Derived, never stored

Both path and bounds are computed and never cached — so they simply can't go stale.

🧲

References over coordinates

By holding the two elements (not their points), the connector tracks them for free as they move.
