What is a Window?

The root of the scene graph — the full-screen stage every other element sits on.

Window · #ff8c42 depth 2 — deepest class extends Container

The Idea

A Window doesn't sit somewhere on the screen — it is the screen.

🎬

Think of a Window as the stage in a theatre — or the blank canvas on an easel. Actors and props get placed onto it, but the stage itself doesn't need a position: it always fills the entire room. That's why a Window always occupies (0, 0, 100, 100) — the whole normalized space — and its constructor takes no bounds. It is the one frame everything else lives inside.

🌳

Root

The top of the tree. A Window has no parent — every other element traces its lineage back up to it.

🖼️

Full bounds

Implicitly (0, 0, 100, 100). The constructor takes no bounds — they're always the full screen.

▶️

Render entry point

to_html() — the single call that turns the whole tree into a standalone file.

  1. Window extends Container, so it can hold children — it's a branch node, not a leaf.
  2. Its bounds are implicitly full — you never pass bounds; it models the whole screen / viewport.
  3. to_html(path, size=None) renders this Window and all descendants to a standalone HTML file.
  4. size=None → a full-screen stage; size=<int> → a fixed size × size square stage.
  5. It is the thinnest subclass in the package — just a constructor plus to_html.

This Module's Contribution

Two members, no surprises — fix the bounds, add a render shortcut.

i_2_window/main.py class Window(Container) 

def __init__(self,
             name: str = 'Window',
             background: RGB | None = None,
             border: Border | None = None) -> None:
    # No bounds parameter — a Window is always (0, 0, 100, 100).
    Container.__init__(self, name=name, background=background,
                       border=border)

def to_html(self, path: str, size: int | None = None) -> None:
    # Render this Window (and its descendants) to a standalone HTML file.
    from f_gui.render.html import RenderHtml
    RenderHtml.to_file(root=self, path=path, size=size)

❌ import at top of file

  • The data model would depend on the renderer at module load.
  • Risks an import cycle: model → renderer → model.

✅ lazy import inside the method

  • from f_gui.render.html import RenderHtml runs only when to_html is called.
  • Breaks the cycle — the model stays renderer-free at load time.

Fun Facts

Five things worth knowing about roots, viewports, and lazy imports.

🌳

Exactly one root

A tree has precisely one root, and it has no parent — every other node traces back to it.

🖼️

"Viewport"

The visible region a UI is drawn into. A Window models the whole one as the range 0–100.

🔁

The lazy-import idiom

Importing inside a method (not at the top of the file) defers the dependency and breaks cycles.

▶️

One entry point

Window is the natural single place to start rendering — render the root and the whole tree follows.

🪶

Deepest, yet simplest

At depth 2 it's the deepest class here, yet also the simplest — it only fixes bounds and adds to_html.
