What is a Container?

An Element that holds children — and quietly keeps the parent/child tree consistent for you.

Container Element + HasChildren

The Idea

A box that owns other boxes — and never lets one box have two owners.

🗂️

A Container is a folder that holds files. Drag a file into another folder and it automatically leaves the first one — you never end up with the same file living in two places. A Container does exactly this for Elements: attach a child somewhere new and its old home is cleaned up for you.

❌ Hand-managing the tree yourself

  • You keep a children list and a parent pointer in every class.
  • Move a child without remembering to remove it from the old parent → the child now sits in two parents.
  • Dangling pointers: child.parent says one thing, the parent's children says another.
  • The bug surfaces far away — during render or traversal, not at the move.

✅ Let the Container own the invariant

  • add_child() auto-detaches the child from its old parent first.
  • Every node has exactly one parent, always.
  • The parent pointer and the children list can never disagree.
  • One place owns the rule, so there's one place to get it right.

The Rules It Enforces

Everything the Container guarantees, in plain terms.

  1. add_child(child) detaches the child from its previous parent first — so attaching it here removes it from there.
  2. Re-adding a child already owned by self is a no-op — the old_parent is self check short-circuits, so there's no duplication.
  3. remove_child(child) clears child.parent — a detached child reports parent is None.
  4. The parent pointer is only ever mutated via HasParent._set_parent() — callers never touch _parent directly.
  5. The children list itself comes from the HasChildren mixin — the Container just wires the parent side onto it.

How It Connects

Container sits one rung above the abstract Element, and is the base of Window.

Element # abstract base — bounds + name + parent └── Container # + HasChildren — this module └── Window # root container, full 0–100 bounds

A Container may hold any Element — another Container, a Label, a Line, a Connector, even a Window. Children nest arbitrarily deep — that's the Composite pattern at work.

This Module's Contribution

The two methods that keep the tree honest — verbatim from main.py.

single owner of the tree-mutation invariant 

def add_child(self, child: Element) -> None:
    # Detach from old parent (if any)
    old_parent = child.parent
    if old_parent is not None:
        if old_parent is self:
            return
        old_parent.remove_child(child=child)
    # Attach to this Container
    child._set_parent(parent=self)
    HasChildren.add_child(self, child=child)


def remove_child(self, child: Element) -> None:
    HasChildren.remove_child(self, child=child)
    child._set_parent(parent=None)

Read the flow: grab the old parent → if it's already us, bail (no-op) → otherwise remove from the old parent → set our pointer → append to our list. Removal is the mirror image: drop from the list, then null the back-pointer.

Fun Facts

Why the design looks the way it does.

🧩

It's the Composite pattern

A Gang-of-Four classic: treat individual objects and compositions of them uniformly, so a Label and a deeply-nested Container are both just "an Element".

🌐

The DOM does this too

JavaScript's appendChild also reparents — moving an existing node detaches it from its old parent. Same semantics as add_child.

🌳

Two tree invariants

Exactly one parent per node, and no cycles. Auto-detach-on-add is what keeps the "one parent" half true without any caller effort.

🎞️

Retained-mode scene graph

The tree is stored, not re-issued every frame. Mutate it once and the structure persists — the opposite of immediate-mode drawing.

🔒

Why _set_parent is private

Encapsulating the back-pointer is what prevents the classic "child in two parents" bug — no caller can set _parent behind the Container's back.
