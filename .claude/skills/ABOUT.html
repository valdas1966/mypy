? What Are Skills?

Skills are packaged prompts that teach Claude how to do specific tasks.

Each skill is a SKILL.md file (YAML frontmatter + markdown instructions) stored in a .claude/skills/ directory. When invoked via /slash-command or auto-triggered by Claude, the skill's content loads into context and Claude follows the instructions using its tools.

They are not code execution — they are instructions that guide Claude's reasoning and tool use. Think of them as reusable playbooks.

~ Pros & Cons

+ Pros

1
Reusable Workflows Package deploy, review, migrate as /commands. Write once, use everywhere.
2
On-Demand Loading Only descriptions stay in context. Full skill loads only when invoked — saves context budget.
3
Auto-Activation Claude can trigger a skill when your request matches its description. Zero friction.
4
Isolated Execution context: fork runs in a subagent — no conversation pollution.
5
Dynamic Context Injection !`gh pr diff` injects live data before Claude sees the prompt.
6
Tool Gating allowed-tools grants specific permissions per skill. Safe read-only modes possible.
7
Team Shareable Project-level skills commit to git. Everyone gets them automatically.
8
Model Override Use Opus for complex analysis, Haiku for quick checks — per skill.

− Cons

1
Not Enforced Skills are instructions, not rules. Claude tries to follow them but may deviate. No guarantees.
2
Context Cost Descriptions consume ~2% of context. Many skills may be excluded at budget limits.
3
Static Dynamic Data !`cmd` runs once before Claude sees the prompt. No live updates during execution.
4
No Skill Chaining A skill cannot invoke another skill. Must chain manually or use subagents.
5
No State Persistence Each invocation starts fresh. No memory between runs unless you write to disk.
6
Unreliable Auto-Trigger Description-based activation is heuristic. Claude may or may not fire the skill.
7
Fork Loses History context: fork runs without conversation history. Only sees skill content.
8
Scope Conflicts Same-named skills at multiple levels: higher priority wins. No merging or inheritance.
