How LLMs Think
Explained for people who know A*

You know search. You write prompts. But what happens inside the model? It's simpler than you think — and it has a big problem you know how to fix.

Step 1

An LLM is a Next-Word Machine

That's it. An LLM reads all the words so far, and predicts the next word. Then it adds that word, and predicts the next one. And again. And again.

How it generates a response

You type: "What is 2 + 3?"

The LLM builds its answer one word at a time:

The answer is ??? ???

It picks the next word by probability:

5
■■■■■■■■■
85%
4
5%
six
3%
...
7%
The answer is 5 ???

It picked "5" (highest probability) → added it → now predicts the next word → picks "." → done.

Think of it as Greedy Best-First Search

At every step, pick the single best-looking option. No backtracking. No lookahead. Just greedy, one step at a time.

Step 2

Easy questions work. Hard questions break.

"2 + 3" is easy — the LLM "knows" the answer instantly. But what about problems that need multiple steps of thinking?

A farmer has 15 apples. He gives ⅓ to his neighbor. Then he buys double what he has left. How many apples does he have now?

This needs 3 steps: (1) find ⅓ of 15 → (2) subtract → (3) double. Each step depends on the previous one. One wrong step ruins everything.

❌ Without reasoning (greedy)
The farmer has 25 apples

The LLM jumps straight to an answer.
Picks the "most likely looking" number.
Wrong. No work shown. No way to check.

✓ With reasoning (chain of thought)
⅓ of 15 = 5
15 - 5 = 10
10 × 2 = 20
Answer: 20 ✓

The LLM generates thinking steps before answering. Each step is a new "word" it predicts.

"Reasoning" = generating thinking tokens before the answer

The LLM literally writes "⅓ of 15 = 5" as output tokens — it thinks by writing. Each thinking step is just more next-word prediction.

Step 3

The Big Problem: Greedy Reasoning Gets Lost

Even with step-by-step thinking, the LLM still picks each step greedily. If it makes a wrong choice at step 2, everything after is garbage. It can't backtrack.

Harder prompt where greedy fails
A bat and ball cost $1.10 together. The bat costs $1.00 more than the ball. How much does the ball cost?

The greedy LLM "reasons":

   Step 1:  "The bat costs $1.00"              ← Feels right, but WRONG framing
      │
   Step 2:  "The ball costs $1.10 - $1.00 = $0.10"  ← Follows logically from wrong step
      │
   Answer: "$0.10"                             ← WRONG (correct: $0.05)

The first step looked right (high probability), so the LLM committed to it. It can't go back and try a different first step. Sound familiar?

This is Greedy Best-First without a heuristic

You already know this problem! It's a search that always picks the locally-best node but never backtracks and has no h(n).

Step 4

LLM Reasoning IS a Search Tree

What if, instead of committing to one reasoning path, the LLM generates multiple possible next steps and explores them like a tree? This is exactly what new models (o1, o3, DeepSeek-R1) do.

The bat-and-ball reasoning tree
                            [ Prompt: bat & ball ]
                           ╱               ╲
                 Step A:                     Step B:
          "bat = $1.00"             "let ball = x,
                 │                       bat = x + $1.00"
                 │                              │
          Step A2:                      Step B2:
    "ball = $0.10"              "x + (x+1) = 1.10"
                 │                              │
          Answer:                        Step B3:
        "$0.10" ✗                    "2x = 0.10"
                                                │
                                         Answer:
                                       "$0.05" ✓

Two branches. Path A looks easier (high probability at step 1) but leads to the wrong answer. Path B looks harder (algebra) but reaches the correct answer.

Now here's the key question:

Which branch should we explore first?

And how do we know when to stop exploring and trust the answer?

Step 5

How They Search Today (Badly)

Beam Search

Keep the top-K "most likely" branches. Drop the rest. 

  [Start]
   ├─ A (prob: 0.7) ← keep
   ├─ B (prob: 0.2) ← keep
   └─ C (prob: 0.1) ← DROP

Problem: "Most likely" ≠ "most correct." Drops good paths that look unlikely. No h(n). No optimality. Just probability.

MCTS (Monte Carlo Tree Search)

Randomly sample many complete paths. Pick the best one. 

  [Start]
   ├─ A → A2 → wrong
   ├─ A → A3 → wrong
   ├─ B → B2 → right!
   ├─ A → A2 → wrong  ← wasted
   └─ A → A4 → wrong  ← wasted

Problem: Random = wasteful. Explores many bad paths. Each "rollout" costs real LLM tokens = real money. No guidance.

Method In Search Terms Optimality? Efficiency?
Greedy Greedy best-first, b=1 None Fast but wrong
Beam (K) Greedy best-first, b=K None K× cost, still no guarantee
MCTS Random sampling Statistical Very expensive (many rollouts)
A* Best-first with h(n) Guaranteed* Minimum expansions*

* With an admissible heuristic — this is the part that needs research!

Step 6

How A* Fixes LLM Reasoning

Apply the same A* you use on grids — but on the tree of reasoning steps.

A* on the bat-and-ball reasoning tree
Node g (effort) h (estimated quality) f = g + h Action
[Start] 0 — — Expand → A, B
Step A "bat=$1" 1 3.0 (looks easy but smells wrong) 4.0 In frontier
Step B "let ball=x" 1 1.5 (structured, algebraic) 2.5 ← lower! Expand first!
Step B2 "x+(x+1)=1.10" 2 0.5 2.5 Expand → B3
Step B3 "$0.05" 3 0 3.0 ✓ Goal reached!

Result: A* found the correct answer by expanding only 4 nodes. It never even explored the wrong branch A — because h(n) told it that branch was less promising.

Step 8

The Whole Picture in 30 Seconds

  1.  LLM = a machine that predicts the next word.

  2.  "Reasoning" = the LLM writes thinking steps as output words
      before giving the final answer.

  3.  Each thinking step is chosen greedily — highest probability.
      No backtracking. No lookahead.

  4.  This is exactly like greedy best-first search without h(n).
      It works on easy problems. It fails on hard ones.

  5.  Current fixes (beam search, MCTS) are like BFS or random walks.
      Expensive. No optimality guarantee.

  6.  A* with a good heuristic would find correct reasoning paths
      with minimum LLM calls (= minimum cost).

  7.  The open research problem: how to build a good h(n)
      for reasoning trees. This is where you come in.
