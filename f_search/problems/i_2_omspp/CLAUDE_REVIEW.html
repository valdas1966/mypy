Code + Design Review: ProblemOMSPP

f_search/problems/i_2_omspp/

1. Correctness

  1. [Low] to_spps() iterates self.goals and creates a new ProblemSPP per goal, passing the same self.grid reference. Correct, but if the grid is None (post-pickle, pre-reload), this will raise ValueError inside ProblemSPP.__init__ when it accesses grid.name. Consider documenting this precondition.
  2. [Info] No validation that goals list is non-empty. An empty goals list produces an empty to_spps() result, which is logically consistent but may cause silent failures downstream.
  3. [Info] h_start, h_goals, norm_h_start, norm_h_goals correctly handle edge cases: zero-guard for trivial grids, 0.0 for single-goal h_goals. Pairwise enumeration in h_goals uses i < j to avoid double-counting.
  4. [Info] to_analytics() correctly extends base dict via ProblemSearch.to_analytics().

2. Readability / Naming

No change needed.
  1. Class name ProblemOMSPP clearly conveys One-to-Many Shortest Path Problem.
  2. Method to_spps() is concise and descriptive.
  3. Constructor parameter names match the mixin expectations exactly.

3. Design (OOP)

  1. [Info] Explicit base-class __init__ calls (ProblemSearch.__init__, HasStart.__init__, HasGoals.__init__) instead of super(). Consistent with the project pattern and safe for the current flat mixin composition. No change needed unless cooperative super() is adopted project-wide.
  2. [Info] Good mixin composition: ProblemSearch + HasStart + HasGoals provides clear separation of concerns with no overlapping methods.

4. Performance

  1. [Info] to_spps() creates k new ProblemSPP objects sharing the same grid reference. Memory-efficient since the grid is not copied. No concern.

8. Architecture

  1. [Info] The class correctly inherits __getstate__ and load_grid from ProblemSearch without overriding. Clean removal of the previous to_light()/to_heavy() pattern.
  2. [Info] Constructor only accepts Grid (not Grid | str), which simplifies the API. Grid loading after pickle is handled by the inherited load_grid() method.

9. Usage & Usefulness

  1. When useful: Multi-destination routing on grids; decomposition into SPP sub-problems for algorithms like KxAStar.
  2. How to use: Construct with Grid, State, list[State], then pass to an OMSPP algorithm or call to_spps() for decomposition.
  3. Not overengineered: Only 46 lines; adds exactly one method (to_spps) beyond what the mixins provide.
  4. Factory coverage: Six factory methods cover common test scenarios (no obstacles, obstacles, all goals, cached, custom).

  2. Add _tester.py: No unit tests exist for ProblemOMSPP. Minimal tests for to_spps() and property access would add safety. Low risk.
