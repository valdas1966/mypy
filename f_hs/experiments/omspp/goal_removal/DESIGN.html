Early-Stop — Architecture
one cold-path hook, not a class per algorithm

How to add "stop after reaching r of k goals" to the four OMSPP algorithms — without slowing the millions of normal runs, and without a parallel class tree.

touches AlgoOMSPP · i_0_base new types 0 hot-path Δ none status DESIGN — awaiting OK
Verdict No dedicated early-stop classes. Add one overridable hook on the base, called at the cold goal-record site (~k×/run). Early-stop fires each algorithm's existing done-condition early — so the per-node hot loop gains zero code. A quota is a param, not a type.

1Your two questions, answered

Q1 · don't tax non-early-stop runs

Granted — and it's the deciding constraint. The fix: the stop decision rides the cold goal site (~k×), never the hot loop header (10⁵–10⁶×). Normal runs execute the identical loop they do today.

Q2 · inheritance with hooks (on_goal, can_terminate)

Right instinct, one caveat. Yes to a base hook (Template-Method — the project convention). No to a per-algo subclass tree. And put the hook at the goal site — a virtual can_terminate() in the loop header is the one shape that breaks Q1.

2Where the check lives — and where it must not

The anti-pattern A virtual while not self.can_terminate(): adds a method call per pop (frame + MRO lookup), paid by every run — the one design that violates Q1. If you want a loop-level gate, set a plain boolean flag at the cold site and read it inline; never dispatch in the header.

3The seam — three small edits, no new files

# AlgoOMSPP base (i_0_base) — cold path, ~k calls/run. Default: never stop.
def _on_goal_reached(self, goal, cost, n: int) -> bool:
    return self._stop is None or not self._stop(n, goal, cost)
