Goal Removal & Early Stop
how few of the k goals do you actually need before INC stops winning?

Hold the OMSPP geometry fixed (k = 200 goals, the pinned i_3 problems) and vary the stopping rule: require only the first r of the k goals, removing the rest. r = 1 is "reach any goal, return that path, stop"; r = k is the full OMSPP we already measure. Sweep r and measure expanded nodes, runtime, memory for kA*-INC vs kA*-AGG — and find the crossover r* where the INC-wins verdict turns on.

Bottom line This sweep probes the OMSPP verdict along a new axis — not goal geometry, but how many goals the query actually requires. The headline is a single curve: AGG/INC cost ratio vs the quota r, with a marked crossover r*. Strong prediction: INC's win erodes — and may invert — as r → 1, because AGG's Φ=MIN front is natively aimed at the nearest goal while plain INC chases a fixed goal order. Best part: because INC is incremental and both algorithms shrink their active set only on-reach under a quota, the entire r-curve is derivable from a single full run per algorithm by snapshotting at each goal-reach — exact, no re-runs. 0What "removing nodes" means here

Scope — we remove goal nodes, not arbitrary graph nodes In OMSPP the only nodes the query owns are the goals. "Removing a node during the search" therefore means dropping a goal from the active set A mid-search — either because it was reached (and we now need one fewer) or because the query no longer requires it. Two flavors, both in scope:
  • Quota / early-stop (primary). Require any r of the k goals. Removal happens on-reach: the r-th reached goal triggers termination; the other k−r are discarded. r=1 is the user's example ("reach the first goal, return that path, stop").
  • Online removal (secondary). An external schedule cancels still-active goals partway through the search (e.g. drop a random subset after N expansions) — the "online goals" setting, where requirements change during planning.

1Removal is the mirror of extend

The parent pipeline's s_4 runs INC in extended mode: start at k=10 and extend() the goal sequence up to k=200 — the active set grows. This experiment is the reflection: start at k=200 and let the active set shrink to r. The same SearchStateSPP bundle (OPEN / CLOSED / g / parent) that makes extend cheap is what makes early-stop free — you simply stop running sub-searches.

2Why the quota r flips the verdict

The crux — early-stop favors the aggregated heuristic At r = 1 the task is "reach any goal" = nearest-goal search. AGG with Φ=MIN is exactly the right algorithm: its front is guided by min over active goals, so the first goal it pops is the nearest one — no wasted exploration. Plain INC, by contrast, commits to a fixed goal order and drives its first sub-search at t₁, which may be the farthest goal. So at small r the usual story can invert: AGG ≤ INC.
RegimeWho is favoredMechanism
r = 1 (any goal) AGG (or INC only if nearest-first) AGG-MIN is natively nearest-goal directed; fixed-order INC may chase a far goal first. Trunk is short relative to total ⇒ AGG's per-node h-cost penalty barely accrues.
small r toss-up The crossover region. Per-node h-cost gap (favors INC) starts to outweigh heuristic-direction advantage (favors AGG). Location of r* is the measured quantity.
r → k (full OMSPP) INC Established: INC ≤ AGG for consistent h. Many all-active nodes ⇒ AGG pays ≈k h/node and carries the k-vector; INC pays 1/node and shares CLOSED across sub-searches.
Hypotheses H1 (anchor): at r=k, INC ≤ AGG on all three metrics (reproduces the main result).
H2 (erosion): AGG/INC cost ratio is monotone increasing in r — the INC advantage grows with the quota.
H3 (inversion): ∃ a crossover r* > 1 with AGG < INC for r < r* (at least for fixed-order INC); nearest-first / online-min ordering pushes r* toward 1.
H4 (memory): the memory ratio crosses at a different r* than runtime — peak OPEN/CLOSED is set on the trunk, so memory favors INC even at smallish r (AGG carries the k-vector from node 1).

Fairness is the whole game here Comparing AGG-MIN (natively any-goal) against fixed-order INC at r=1 is the interesting but unfair contrast — it makes AGG look great for a reason that has nothing to do with aggregation. So we always report INC in all three orderings. online-min INC is the apples-to-apples comparator: same any-goal greediness as AGG-MIN, but incremental h. If online-min INC still loses to AGG at small r, the win is genuinely AGG's structure; if it ties, the early-stop advantage was just heuristic direction, recoverable by reordering. This is the time–memory / baseline-fairness neutralization the strategy notes call for.

4The cheap derivation — snapshot at every goal-reach

We do not need to run the search once per r. One full run per algorithm, snapshotting the cumulative counters at each on_goal event, yields the cost-to-reach-r for all r at once — exactly how the parent s_4 snapshots cumulative counters per k-checkpoint, but finer (per reach, not per k-batch).

AlgorithmIs "truncate the full run at the r-th reach" exact for a real quota-r run?
INC (any order) Exact — always. A quota-r run is literally the first r sub-searches of the full run; sub-searches r+1…k never start, and they never altered the earlier trajectory. Counters, peak memory, and even wall-clock up to the r-th reach are identical.
AGG (on-reach quota) Exact. Under a quota you cannot drop a goal before it is reached (you don't know which r you'll get), so the active set shrinks only on-reach — the same shrink the full run already performs. Trajectory up to the r-th pop is identical ⇒ truncation is exact.
AGG / INC (online removal) Not exact. Cancelling a not-yet-reached goal changes Φ (AGG) or the remaining sub-search set (INC) ⇒ the trajectory diverges. Requires real runs (study B).
One honest caveat — wall-clock vs counters Counters and peak memory truncate exactly. Wall-clock at a checkpoint is the cumulative elapsed up to that reach; a real early-stop run additionally skips end-of-run cleanup, so derived runtime is an upper bound that is tight to within the teardown cost. Validation: s_7 runs a handful of real quota-r runs (r ∈ {1,10,50}) and checks the derived curve matches; report the residual.
