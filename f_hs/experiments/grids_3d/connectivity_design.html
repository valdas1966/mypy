Connectivity Redesign — Enabling 8-Neighbors

A design plan for f_ds/grids: make connectivity a first-class, swappable policy — 4↔8 in 2D today, 6/18/26 in 3D soon.

f_ds/grids · f_core/mixins/has/row_col · f_hs/problem · plan only — no code changed · 2026-06-21

TL;DR. Today the grid metric is welded into one core identity mixin (HasRowCol: position and neighbors and Manhattan distance) and the search hot-path assumes integer edge weights. 8-connectivity breaks both: it needs a different neighbor set, a different heuristic, and a √2 (non-integer) move cost. The fix is one idea — extract a Connectivity policy that owns offsets + cost + heuristic + corner-cutting, inject it into the grid, and default it to 4-conn so nothing existing changes. The same seam yields 6/18/26 in 3D by swapping the policy, not rewriting the grid.

01What we're changing — and the two walls

Two facts in the current code make "just add diagonals" impossible without a design change. Both are real and both are quoted below.

Wall 1 — the metric is welded to identity

One core mixin HasRowCol conflates three concerns:

identity (row, col), key, eq/hash
topology neighbors() → hard-coded 4-conn
metric distance() → hard-coded Manhattan

Because topology + metric live inside identity, there is no seam to vary them — and the mixin is reused by CellBase, CellMap, every grid, and f_hs.

Wall 2 — the integer-weight invariant

ProblemSPP.w() returns int and documents a deliberate hot-path assumption:

"Integer-weight assumption: the framework treats edge costs as int in its hot path … Fractional weights would require a separate weighted variant (not shipped)."

A cardinal move is 1. A diagonal move is √2 ≈ 1.414 — not an integer. So 8-conn collides with the invariant head-on. This is the decision in §03.

02The principle — split identity · topology · metric

The whole redesign is one move: pull topology and metric out of the identity mixin into a Connectivity policy, and inject that policy into the grid. Identity (HasRowCol / future HasXYZ) keeps only position + key. The grid owns a policy; the policy owns "how cells connect and what moves cost."

BEFORE — one welded mixin
HasRowCol {
  id row, col, key
  topo neighbors() 4-conn
  met distance() Manhattan
}

topology & metric trapped inside identity

AFTER — identity stays pure
HasRowCol { id row, col, key }

Connectivity {
  topo offsets(), neighbors(cell, grid)
  met cost(a,b), heuristic(a,b)
  rule corner-cut check
}

grid holds a swappable policy

03The cost decision (Wall 2) — √2 vs the int hot-path

8-conn needs a diagonal cost of √2. The framework's hot path is integer. Two clean ways out — the policy owns the choice, so it's localised:

OptionCardinal / DiagonalHeuristicHot pathTrade-off
A · scaled-int octile
recommended now		 K=10000 / round(K√2)=14142 octile, same scale unchanged — stays int order/tie structure exact; costs differ from MovingAI floats by a constant factor (irrelevant for INC/AGG node counts)
B · float weights
benchmark-fidelity path		 1.0 / √2 float octile audit needed (priority, g) matches released MovingAI/voxel float costs exactly; revises the documented int invariant — a core stance change
Recommendation: ship Option A (scaled-int octile) first — it preserves the framework's stated integer invariant with zero hot-path disruption, and for the INC-vs-AGG line only ordering & tie structure matter, which A reproduces exactly. Move to Option B only when/if you decide to inherit external float optimal costs (the "Euclidean / benchmark" philosophy). Either way the choice is confined to one Connectivity subclass.

04Proposed architecture

A small policy hierarchy, dimension-agnostic at the base. The grid composes one (default = 4-conn); the problem/heuristic read cost & h from it.

ConnectivityBase  (abstract: offsets, cost, heuristic, is_legal_move)
 ├─ Connectivity4  2D now 4 offsets · all cost 1 · Manhattan · (default)
 ├─ Connectivity8  2D now 8 offsets · 1/√2 (scaled) · octile · no-corner-cut
 ├─ Connectivity6   3D soon faces · Manhattan-3D
 ├─ Connectivity18  3D soon +edges
 └─ Connectivity26  3D soon +corners · voxel-octile

Injection — three small delegations

GridMap

holds connectivity (default Connectivity4); neighbors() delegates to connectivity.neighbors(cell, self).

ProblemGrid.w

returns grid.connectivity.cost(parent, child) instead of constant 1.

heuristic

A*'s h reads grid.connectivity.heuristic(s, goal) instead of s.distance(goal).

Plus a one-line fix to GridBase.norm_distance to ask the policy for its max distance. HasRowCol.neighbors/distance stay in place (now vestigial) for backward-compat, deprecated later.

05Neighbor offsets & the corner-cutting rule

4-conn vs 8-conn (2D)

cell cardinal · cost 1 diagonal · cost √2

No-corner-cutting rule

The diagonal A→B is legal only if both flanking cardinals are free. One blocked flank (✗) ⇒ the diagonal is rejected. This lives in Connectivity8.is_legal_move and needs the grid's validity oracle — which is why neighbors(cell, grid) takes the grid.

Consequence (already established): under strict no-corner-cutting, 8-conn adds no reachability over 4-conn — only cheaper cost. So the default flip is safe; only path costs change.

06Built for 3D from day one

The policy is dimension-agnostic: it generates offsets over a position type and returns cost/heuristic. 3D = a new identity mixin (HasXYZ) + new Connectivity{6,18,26} subclasses. The grid and problem code do not change. Mirror, don't fork.

Consistency pairs (don't cross them)

Hold the metric philosophy constant across dimensions, or 2D↔3D comparisons confound dimension with metric:

  • L1 column: Connectivity4 ↔ Connectivity6 — self-generated costs.
  • Euclidean column: Connectivity8 ↔ Connectivity26 — can inherit benchmark float costs (Option B).

The HasRowCol welding is the anti-pattern to not repeat in HasXYZ.

07Migration — backward-compatible by construction

Default Connectivity4 reproduces today's behaviour byte-for-byte (4 offsets, all cost 1, Manhattan). Every existing test and every hard-coded optimal cost stays green. 8-conn is opt-in.

  1. Extract ConnectivityBase + Connectivity4 in f_ds/grids (new connectivity/ module). Move the 4 offsets + Manhattan into Connectivity4. core area — needs your OK; it's foundational with ≥2 reusers.
  2. Inject into GridMap: add connectivity=Connectivity4() default; route neighbors() through it. No public signature change.
  3. Route cost & heuristic: ProblemGrid.w → connectivity.cost; A* h → connectivity.heuristic; fix norm_distance.
  4. Add Connectivity8 (Option A scaled-int octile + no-corner-cut). New _tester.py: diagonal legality, octile values, flank rejection.
  5. Regenerate the 8-conn ground truth for the experiment fixtures by re-running the internal solver (costs are self-generated — mechanical, not a research redo).
  6. Update CLAUDE.md in every touched folder (row_col, grids/grid, grids/cell, f_hs/problem) per for_claude_md.md.
Gate. Step 1 edits a core mixin area reused by f_ds grids and f_hs (rule-of-three: ≥2 reusers). Per project convention this is proposed, not silently built — this document is that proposal; it waits for your OK before any code is written.
