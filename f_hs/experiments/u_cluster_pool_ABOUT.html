Candidate-Location Pool Sampler
one sampler — START pool & GOAL pool are just parameters

The shared, grid-based building block under every heuristic-search experiment: sample random ClusterDiamond locations across grid maps and stream their metadata to a CSV. What used to be two near-identical scripts (s_0_clusters_start, s_1_clusters_goals) is one function called with different parameters.

module f_hs/experiments/u_cluster_pool.py used by mospp · omspp · goal_distance tests 5 passing shape diamond (Manhattan ball)
In one line Sample n random diamonds of radius steps (each with ≥ min_cells cells) per grid, and write domain, map, center_row, center_col, steps, cells rows to a CSV on Drive. START pool = steps=0, min_cells=1; GOAL pool = steps≥1, min_cells=k. Same code, different params.

1Start pool vs goal pool — the same sampler

START pool · steps=0, min_cells=1

Each "cluster" is a single cell — a candidate place a search can begin. Downstream, one start is paired with one goal region.

GOAL pool · steps≥1, min_cells=k

Each cluster is a diamond region of ≥ k cells, so k goals can be drawn from it later. Radius (steps) controls how spread the goals can be.

2API — two entry points

FunctionRoleDrive?
sample_pool(grids, steps, min_cells, n, max_tries) pure core — yields one metadata-row dict per sampled clusterno
generate_pool_csv(path_grids_pkl, path_csv, steps, min_cells, n, seed, max_tries, drive) orchestration — drive.read_pickle grids → sample → drive.upload_rows CSVyes

Pure logic (sample_pool) is separated from Drive I/O (generate_pool_csv) so the core is unit-testable on an in-memory grid and the per-experiment s_N wrappers stay one call long. The generic Drive plumbing (pickle / CSV / temp-files) now lives on the Drive facade — read_pickle / upload_pickle / upload_rows — so it is shared by every experiment script, not re-implemented here.

⚠ Reproducibility & re-runs Sampling uses the process-global random module (GridMap.random.cells → random.sample), so seed calls random.seed. Re-running a seeded wrapper regenerates the pool and would cascade to the downstream pinned artifacts (i_2 pairs / i_3 problems). Re-run only to deliberately regenerate the whole pipeline.

4Where it sits

Scope: grid-map experiments only. Puzzle / road-network benchmarks would need their own pool samplers — this one is intentionally grid-specific.
