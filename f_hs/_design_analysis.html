1 Should f_hs Inherit from f_cs?

✅ Verdict: YES — Inherit from f_cs

The benefits (lifecycle hooks, timing, type framework) outweigh the minor baggage. Rebuilding from scratch would produce something nearly identical to what f_cs already provides.

The core question: does Algo(ProcessIO[Problem, Solution]) provide enough value to justify the inheritance chain?

The Inheritance Chain You'd Get

Equatable ↓ Comparable ↓ HasName — name, __str__, comparison, hash ↓ ProcessBase — run(), _run_pre/_run/_run_post, elapsed timing ↓ ProcessIO[Problem, Solution] — input property, output capture ↓ Algo[Problem, Solution] — problem property, cls_stats, elapsed→stats ↓ AlgoHS[Problem, Solution] — YOUR new base with event hooks

⚠️ Why Not Rebuild From Scratch?

Rebuilding lifecycle + timing + generic typing from scratch would produce something nearly identical to what f_cs provides — just to avoid an unused self.input alias. The engineering cost doesn't justify the cosmetic purity.

3 Does Heuristic Search Need a Solution Class?

Three real-world patterns exist for returning algorithm results. Let's evaluate each for analysis-focused heuristic search:

4 Three Design Patterns Compared

Pattern A: Solution Object (f_cs / scipy style)

# Algorithm produces a self-contained result object
solution = astar.run()        # returns Solution
solution.path                 # the path
solution.stats.explored       # counter
solution.stats.events         # event log

Pattern B: Algo-as-Result (sklearn style)

astar = AStar(problem)
astar.run()          # mutates self
astar.path           # on the algo

# No structure

❌ Pattern B's Fatal Flaw

After astar.run(), the AStar object holds the entire search state — frontier, explored set, parent pointers, g/h dictionaries. On a 100×100 grid, that's tens of thousands of entries. You'd carry all that dead weight just to read astar.stats.explored.

Solution objects carry only what matters: the result.

✅ Verdict: YES — Keep the Solution Class

Pattern A wins for analysis-focused work. Solution objects are self-contained, serializable, lightweight, and enable clean comparison across runs.

5 Should Stats Live Inside Solution?

What "Stats" means today (f_search)

elapsed: float discovered: int explored: int relaxed: int

Just counters. A flat summary. Fine for f_search.

What "Stats" needs to mean in f_hs

elapsed: float discovered: int explored: int relaxed: int events: list[dict] — NEW to_dataframe() — NEW

Two Design Options

✅ Option A: Events Inside Stats

Solution ├―― path: Path ├―― is_valid: bool └―― stats: StatsHS ├―― elapsed, discovered, explored (summary) └―― events: list[dict] (detail)
  • One object for all execution data
  • Counters and events can't go out of sync
  • Counters ARE aggregates of events — same concept, different granularity
  • Consistent with f_cs pattern

Option B: Events as Sibling

Solution ├―― path: Path ├―― is_valid: bool ├―― stats: StatsHS (counters only) └―― events: EventLog (separate object)
  • Artificial separation of related data
  • Synchronization burden (counters must match events)
  • Two objects to pass around
  • Breaks f_cs convention
💡 The Key Insight

Counters are aggregates of events. They're the same data at different granularity:

# The events:
events = [
    {'type': 'discover', 'state': (0,0), 'g': 0, 'h': 5},
    {'type': 'explore',  'state': (0,0)},
    {'type': 'discover', 'state': (0,1), 'g': 1, 'h': 4},
    ...
]

# The counters are just:
discovered = sum(1 for e in events if e['type'] == 'discover')
explored   = sum(1 for e in events if e['type'] == 'explore')

Putting them in the same object reflects the truth: counters summarize events.

✅ Verdict: YES — Stats Inside Solution, Events Inside Stats

Keep the f_cs pattern. Stats is the execution record. Counters + events = same data, different granularity.

6 The Key Innovation: Opt-In Event Logging

⚠️ The Performance Problem

On a 1000×1000 grid, A* might explore 500K states. Each event = a dict with 5–6 fields. That's ~500K dicts = significant memory.

When running batch benchmarks (10K problems), you want counters only.
When analyzing one specific run, you want the full event log.

Solution: is_logging Flag

class StatsHS(StatsAlgo):

    def __init__(self, is_logging: bool = False) -> None:
        super().__init__()
        self._discovered: int = 0
        self._explored: int = 0
        self._relaxed: int = 0
        self._is_logging: bool = is_logging
        self._events: list[dict] = []

    def log(self, event: dict) -> None:
        """Log an event. No-op when logging is disabled."""
        if self._is_logging:
            self._events.append(event)

Two Modes, Same Algorithm Code

⚡ Fast Mode (benchmarks)

algo = AStarHS(problem)
# is_logging=False (default)
# stats.log() is a no-op
# Zero event overhead
solution = algo.run()
print(solution.stats.explored)

🔎 Analysis Mode (single run)

algo = AStarHS(problem,
              is_logging=True)
solution = algo.run()
df = solution.stats.to_dataframe()
# Full event-by-event trace

The algorithm code is identical in both modes. It always calls self._stats.log(event). The flag controls whether events are stored or silently discarded.

7 Final Recommended Architecture

Inheritance Map

f_cs (REUSE — inherit from these) ┌──────────┬─────────────┬──────────────┐ StatsAlgo ProblemAlgo SolutionAlgo Algo ↓ ↓ ↓ ↓ f_hs (BUILD NEW — inherit from f_cs) ↓ ↓ ↓ ↓ StatsHS ProblemHS SolutionHS AlgoHS counters generic path + cost event hooks + events successors + export

Class Responsibilities

ClassInherits FromAdds
StatsHS StatsAlgo Counters (discovered, explored, relaxed), event log (opt-in), to_dataframe()
ProblemHS ProblemAlgo Abstract successors(state), not tied to grid
SolutionHS SolutionAlgo[ProblemHS, StatsHS] path, g_goal
AlgoHS Algo[ProblemHS, SolutionHS] Event hooks: _on_discover, _on_explore, _on_relax, _on_goal_reached

StatsHS — The Heart

class StatsHS(StatsAlgo):
    """
    ========================================================================
     Stats for Heuristic-Search with optional Event Logging.
    ========================================================================
    """

    def __init__(self, is_logging: bool = False) -> None:
        super().__init__()
        self._discovered: int = 0
        self._explored: int = 0
        self._relaxed: int = 0
        self._is_logging: bool = is_logging
        self._events: list[dict] = []

    def log(self, event: dict) -> None:
        if self._is_logging:
            self._events.append(event)

    def to_dataframe(self):
        import pandas as pd
        return pd.DataFrame(self._events)

AlgoHS — Event Hooks

class AlgoHS(Algo[ProblemHS, SolutionHS]):
    """
    ========================================================================
     Base Algorithm for Heuristic Search with Event Logging.
    ========================================================================
    """

    # No-op hooks — zero cost when not overridden
    def _on_discover(self, state) -> None: pass
    def _on_explore(self, state) -> None: pass
    def _on_relax(self, state) -> None: pass
    def _on_goal_reached(self, state) -> None: pass

AStarHS — Hooks Emit Structured Events

class AStarHS(AlgoSPP_HS):

    def _on_discover(self, state) -> None:
        self._stats.discovered += 1
        self._stats.log({
            'type':  'discover',
            'state': state.key,
            'g':     self._data.g(state),
            'h':     self._data.h(state),
            'f':     self._data.f(state),
            'step':  self._stats.discovered,
        })

    def _on_explore(self, state) -> None:
        self._stats.explored += 1
        self._stats.log({
            'type':  'explore',
            'state': state.key,
            'step':  self._stats.explored,
        })
