Why threads fail here
  1. GIL blocks CPU parallelism. Python's Global Interpreter Lock allows only one thread to execute bytecode at a time.
  2. Threads add overhead without gain. Context switching + GIL contention makes threads slightly slower than sequential for CPU work.
  3. Processes bypass the GIL. Each process has its own interpreter and GIL, enabling true parallel execution.

Why peak < 12x on 12 cores
  1. Process spawn overhead. Creating 12 processes + pickling data costs ~0.5s.
  2. Uneven chunks. 2M items / 12 workers = 166,667 each, but prime density varies — some chunks finish faster.
  3. WSL2 overhead. Virtualization layer adds scheduling latency vs bare-metal Linux.
  4. Amdahl's Law. Sequential portions (splitting, collecting, summing) limit theoretical max speedup.

Key Insights

  1. Threads are useless for CPU-bound Python. GIL prevents parallelism; threads add overhead (0.97x = slower than sequential).
  2. Processes give real speedup. 1.76x with 4 workers, 2.87x with 12 workers on CPU-bound work.
  3. Sweet spot = physical core count. Peak at 12 workers on 12 cores. Beyond that, OS scheduling overhead degrades performance.
  4. Diminishing returns after ~8 workers. Going from 8→12 workers gains only 0.45x more speedup (2.42x→2.87x).
  5. 1 worker is worse than sequential. Process spawn + IPC overhead without parallelism benefit (0.88x).
  6. 2x cores is wasteful. 24 workers = 2.06x, worse than 6 workers = 2.25x. More processes = more context switching.
  7. Threads are for I/O-bound work. Use use_processes=False only for network calls, file I/O, API requests.

Practical Guidelines

When to use what
WorkloadModeWorkersWhy
CPU-bound (math, parsing, hashing) use_processes=True os.cpu_count() Bypasses GIL, true parallelism
I/O-bound (HTTP, file, DB) use_processes=False 10–50+ GIL released during I/O waits
Mixed (API call + parse response) use_processes=False 2x–4x cores I/O dominates; threads are simpler
Tiny input (<100 items) Either 2–4 Overhead may exceed task time
Rule of thumb
  1. CPU-bound: workers = os.cpu_count(), use_processes=True.
  2. I/O-bound: workers = 20 (adjust by latency), use_processes=False.
  3. Never exceed 2x cores for CPU-bound work — diminishing returns guaranteed.
  4. Always benchmark your specific workload. These numbers are for prime counting; your task may differ.
