Code + Design Review

f_core / processes
  1. Modules reviewed: i_0_base, i_1_input, i_1_output, i_2_io, i_3_parallel, __init__.py
  2. Deep-inspected: HasName, ValidatableMutable, Group

1. Correctness

No bugs found.
  1. Template Method lifecycle is correct across all levels: _run_pre → _run → _run_post.
  2. ProcessIO.run() explicitly delegates to ProcessOutput.run(self), correctly resolving the MRO ambiguity.
  3. ProcessParallel clamps workers, handles empty input, preserves chunk ordering, and collects errors correctly.
  4. Timing attributes (elapsed, _time_start, _time_finish) are reset on each run() call.

2. Readability / Naming

No change needed.
  1. Class names follow a clear progression: Base → Input → Output → IO → Parallel.
  2. Folder naming (i_0_, i_1_, i_2_, i_3_) reflects inheritance depth accurately.
  3. Lifecycle hook names (_run_pre, _run, _run_post) are intuitive and consistent.

3. Design (OOP)

Clean design.
  1. Template Method pattern is correctly applied: base defines lifecycle, subclasses override _run().
  2. Multiple inheritance in ProcessIO cleanly combines orthogonal capabilities (input + output).
  3. Composition over subclassing in ProcessParallel: users pass a func instead of subclassing.
  4. Mixin use is appropriate: HasName and ValidatableMutable add focused capabilities.
  5. Explicit parent calls (e.g., ProcessInput.__init__) are correct for diamond inheritance.

4. Performance

No change needed.
  1. Base classes add negligible overhead (two time() calls per run()).
  2. ProcessParallel correctly uses forkserver on Linux/macOS to avoid fork-safety issues.
  3. Worker clamping prevents unnecessary thread/process creation.

5. Typing

Minor issue found.
  1. i_0_base/main.py:33 — elapsed property declares return type int but returns None before run() is called. Should be int | None.
  2. All other type annotations are correct and complete across all modules.
  3. Generic type parameters flow correctly through the inheritance chain.

6. Documentation

No change needed.
  1. All classes and methods have docstrings following the ==== separator convention.
  2. Inline comments in ProcessParallel._run() clearly separate each phase.

7. Modularity

No change needed.
  1. Each class in its own folder with standard file layout (main.py, _factory.py, _tester.py, __init__.py).
  2. Package __init__.py exports only the two most-used classes, keeping the API surface small.
  3. Benchmarks isolated in i_3_parallel/benchmark/ subfolder.

8. Architecture

Minor inconsistency.
  1. i_0_base/__init__.py:1 — Imports ProcessBase from _factory.py instead of main.py. 
    # current
from f_core.processes.i_0_base._factory import ProcessBase, Factory

  2. i_2_io/__init__.py:1 — Same pattern: imports ProcessIO from _factory.py instead of main.py.
  3. i_1_input and i_1_output correctly import from main.py.
  4. Both work correctly but deviate from the project convention documented in CLAUDE.md.

9. Usage & Usefulness

  1. How to use: pick the right level (Base/Input/Output/IO/Parallel), override _run(), call run().
  2. When useful: any task that benefits from lifecycle hooks, timing, or typed I/O contracts (ETL, transforms, algorithms, batch processing).
  3. Overengineered? No. Each level adds exactly one capability. The hierarchy is shallow (4 levels max) and each class is small.
  4. Design strength: consistent API across all levels — always call run(), always get elapsed.
