HasName — the regression, visualized

Why HasName is quietly broken, and what it should look like. Generated as a decision aid — not a CLAUDE.* doc.

1 · The idea: the Has* family gives objects an identity

Every Has* mixin picks one value as the object's key, then derives equality, ordering, hashing & repr from it. That's the whole point of the family. Three siblings still do it — one no longer does.

HasKey identity

key = the key
== < hash repr ✅

HasRowCol identity

key = (row, col)
== < hash repr ✅

HasName regressed

key = name — gone
== < hash ❌   repr ✅

HasRowsCols regressed

key = (rows,cols) gone
== < hash ❌

All four are documented as identity mixins. Only HasKey and HasRowCol still are in code. This page is about HasName (the highest-impact one).

2 · What it IS vs what it SHOULD be

NOW · broken current code

class HasName(HasRepr):           # only repr!

    def __init__(self, name='NoName'):
        self._name = name

    @property
    def name(self): return self._name

    def __str__(self): return self.name
    # NO key
    # NO __eq__ / __lt__ / __hash__

Inherits nothing that gives identity. Two same-named objects fall back to Python's default object-identity equality.

SHOULD BE mirror of HasKey

class HasName(Comparable, Hashable, HasRepr):

    def __init__(self, name='NoName'):
        self._name = name

    @property
    def name(self): return self._name

    @property
    def key(self): return self._name  # ← identity

    def __str__(self): return self.name

One added base + one key property → equality, ordering, hashing all come for free (exactly like HasKey).

Why this is the fix, not a new feature: HasKey = key on an explicit key, HasName = key on the name. They should be twins. Someone deleted HasName's key + bases, leaving only repr.

5 · The decision

Option A · recommended restore identity

  • Make HasName(Comparable, Hashable, HasRepr) + key→name.
  • Fixes Group, matches docs & ClusterGrid.
  • Symmetric with the HasKey change we just made.
  • Trade-off: HasName equality changes from identity → by-name (affects set/dict use across f_ds).

Option B keep lightweight

  • Leave HasName as name + repr only.
  • Rewrite its CLAUDE.md + ProcessBase docs to drop identity claims.
  • Fix the broken Group.key() caller separately.
  • Downside: breaks the family symmetry; consumers that want name-identity stay unserved.
