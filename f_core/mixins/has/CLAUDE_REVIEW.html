Code + Design Review

f_core / mixins / has
  1. Modules reviewed: children, key, name, parent, row_col, rows_cols, __init__.py
  2. Deep-inspected: Comparable, Hashable, Equatable, RectLike

1. Correctness

Minor issue found.
  1. parent/main.py:15 — parent: Self = None should be parent: Self | None = None. The current signature does not reflect that None is a valid default.
  2. row_col/main.py:87-91 — is_within() will raise TypeError if called without rect and without all four min/max params, since None <= self.row fails. No guard present.
  3. All other methods are correct: neighbors() filters negatives, distance() uses abs, path_from_root() terminates on None.

2. Readability / Naming

No change needed.
  1. Class names are clear adjective-style mixins: HasKey, HasName, HasRowCol, etc.
  2. Property names match the mixin name: HasKey.key, HasName.name, HasParent.parent.
  3. All docstrings follow the ==== separator convention consistently.

3. Design (OOP)

Clean design.
  1. Mixin composition is the correct pattern — each mixin adds exactly one capability.
  2. HasKey, HasRowCol, HasRowsCols correctly extend Comparable + Hashable and implement key.
  3. HasChildren, HasName, HasParent are standalone — no unnecessary base classes.
  4. HasName does not extend Comparable/Hashable, unlike HasKey. This is intentional — HasName is purely a property mixin, not an identity mixin.
  5. parent/_factory.py:5 — ParentWithName helper class in _factory.py is a good pattern for testing mixins that need additional capabilities.

4. Performance

No change needed.
  1. All mixins are lightweight property holders with O(1) access.
  2. HasRowCol.neighbors() creates 4 objects per call — acceptable for grid traversal.
  3. HasParent.path_from_root() is O(depth) — correct and unavoidable.

5. Typing

Minor issues found.
  1. parent/main.py:15 — parent: Self = None should be parent: Self | None = None.
  2. parent/main.py:24 — Return type HasParent | None is correct but could use Self | None for consistency with the constructor.
  3. name/main.py:14 — Default 'NoName' in code vs 'None' in some CLAUDE.md docs. Code is correct, docs should match.
  4. All other type annotations are correct and complete across all modules.

6. Documentation

No change needed.
  1. All classes and methods have docstrings following the ==== separator convention.
  2. Inline comments above properties (# Object's Row) are consistent.

7. Modularity

No change needed.
  1. Each mixin in its own folder with standard file layout (main.py, _factory.py, _tester.py, __init__.py).
  2. Package __init__.py exports 4 most-used classes; 2 structural mixins require direct import.
  3. No circular dependencies between the 6 submodules.

8. Architecture

Minor inconsistencies.
  1. name/__init__.py:1 — Imports from ._factory instead of .main. Convention is to import class from main.py. 
    # current
from ._factory import HasName, Factory

# convention
from .main import HasName
from ._factory import Factory
  2. parent/__init__.py:1 — Same pattern: imports HasParent from _factory.py instead of main.py.
  3. children, key, row_col, rows_cols correctly import from main.py.
  4. __init__.py — HasChildren and HasParent are not exported. This is a deliberate design choice (less commonly used), not a bug.

9. Usage & Usefulness

  1. How to use: inherit from one or more Has* mixins, call their __init__ explicitly in your constructor.
  2. When useful: any class needing standard identity, naming, positioning, or tree structure. Used extensively across f_graph, f_ds, f_gui, f_search.
  3. Overengineered? No. Each mixin is 20-40 lines. No unnecessary abstraction.
  4. Design strength: single-responsibility mixins that compose cleanly via multiple inheritance.
