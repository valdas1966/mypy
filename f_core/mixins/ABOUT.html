What is a Mixin?

A small, focused class that adds one capability — mix it in, get superpowers.

1. The Idea

🧩 Think LEGO. A mixin is a single LEGO piece. It doesn't build a house on its own — but snap a few together, and you get exactly the house you need. No more, no less.

Without mixins: deep inheritance

class Animal:
    ...
class FlyingAnimal(Animal):
    ...
class SwimmingFlyingAnimal(FlyingAnimal):
    ...  # where does it end?
Each new combo = a new class in an ever-deeper tree.

With mixins: flat composition

class CanFly:
    def fly(self): ...

class CanSwim:
    def swim(self): ...

class Duck(CanFly, CanSwim):
    ...  # done.
Pick capabilities. Compose. Ship.

The 3 Rules of a Mixin

  1. Single responsibility — does exactly one thing.
  2. No standalone use — meaningless alone, powerful when composed.
  3. No __init__ side effects — plugs into any constructor cleanly.

Equality & Ordering
Equatable
gives: __eq__
Equality via abstract key property. Two objects are equal if their keys are equal.
key → SupportsEquality
Comparable
gives: __lt__ __le__ __gt__ __ge__
Full ordering. Extends Equatable — one key property drives all 5 operators.
key → SupportsComparison
Hashable
gives: __hash__
Enables sets and dict keys. Reuses the same key from Equatable — guarantees a == b → hash(a) == hash(b).
__hash__ = hash(self.key)
Size & Validity
Sizable
gives: __bool__ via __len__
Truthiness from size. Non-empty = truthy. Subclass implements __len__.

Wraps a dict with __getitem__, __setitem__, __contains__, __iter__, keys(), values(), items().

Immutable validity flag set at construction. if obj: checks validity.
__bool__ → self._is_valid

Extends Validatable with mutation. Objects can flip validity at runtime.
extends Validatable
Has* — Property Injection
HasKey[Key]
gives: typed generic key
Generic identity. Provides concrete key property that satisfies Comparable + Hashable.
extends Comparable, Hashable
HasName
gives: name, __str__
String name as identity label. Default 'NoName'. String representation returns the name.
name → str
HasRowCol
gives: row, col, neighbors(), distance()
2D grid position. Key is (row, col) for row-major ordering. Manhattan distance and 4-neighbor discovery.
extends Comparable, Hashable
HasRowsCols
gives: rows, cols, shape(), is_within()
2D dimensions. Represents grid shape. __len__ returns area (rows × cols).
shape() → (rows, cols)

gives: children list, add_child()
One-to-many parent→child relationship. Maintains a list of children.
children → list[Self]

gives: parent ref, path_from_root()
Child→parent link. Walks up the chain to reconstruct the full path from root.
path_from_root() → list[Self]

4. Composition in Action

How RGB uses mixins

class RGB(HasName, Comparable):

    @property
    def key(self) -> tuple[float, float, float]:
        return (self.r, self.g, self.b)
↓
Result: RGB gets name, __str__, __eq__, __lt__, __le__, __gt__, __ge__ — all from 2 lines of inheritance + 1 property.

5. Fun Facts & History

🍦
Named After Ice Cream
The term "mixin" comes from Steve's Ice Cream Parlor in Somerville, Massachusetts (1973). Customers chose a base ice cream flavor, then "mixed in" toppings like nuts, cookies, or candy. Gilad Bracha and William Cook borrowed the metaphor for OOP in their 1990 paper. Pick a base class, mix in capabilities.
🏛️
Born in Lisp (1974)
Mixins first appeared in Flavors, an early object system for Lisp at MIT. Flavors literally named classes after ice cream flavors — you'd combine "vanilla" with "chocolate-chip" and "nut" flavors to build your object. The joke stuck; the pattern survived.
💎
The Diamond Problem
Multiple inheritance creates the "diamond problem" — if A inherits from B and C, and both B and C inherit from D, which D does A get? Python solves this with MRO (Method Resolution Order) using the C3 linearization algorithm. Mixins sidestep the issue by being thin and non-overlapping.
🔑
One Property to Rule Them All
In this codebase, a single abstract key property unlocks 5 operators (==, <, ≤, >, ≥) + hashing. Define key once → get Equatable + Comparable + Hashable for free. That's the power of mixin composition.
🌍
Mixins Are Everywhere
Ruby uses mixins as its primary code reuse mechanism (modules). Dart, Scala, Kotlin, and Swift all support mixin-like traits. Even JavaScript got in on it — ES6 class expressions enable mixin patterns. Python's collections.abc module is essentially a mixin library.
⚖️
Mixin vs. Trait vs. Interface
Interface = contract only (no code). Trait = contract + code (Rust, Scala). Mixin = trait + state. Python mixins can carry instance data, making them more powerful than traits — but also more dangerous if you break the single-responsibility rule.
