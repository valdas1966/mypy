Code + Design Review

f_core / mixins
  1. Modules reviewed: equatable, comparable, hashable, sizable, dictable, validatable, validatable_mutable, has/ (6 submodules), __init__.py
  2. Deep-inspected: SupportsEquality, SupportsComparison, RectLike, Sized

1. Correctness

Minor issues found.
  1. dictable/main.py:18 — data: dict[K, V] = None should be data: dict[K, V] | None = None. Current type hint does not reflect that None is a valid default.
  2. dictable/main.py:50 — get() return type is V | None but default param type is V = None. Should be default: V | None = None.
  3. equatable/main.py:35 — __eq__ accesses other.key without checking isinstance. Will raise AttributeError when compared with non-Equatable. The CLAUDE.md states this is intentional, but note it differs from Python convention of returning NotImplemented.
  4. comparable/main.py:32 — Same pattern: __lt__ accesses other.key without guard. Consistent with Equatable design, but documented for awareness.
  5. All other methods are correct: Hashable.__hash__ delegates to key, Sizable.__bool__ delegates to __len__, ValidatableMutable correctly calls super().__init__.

2. Readability / Naming

No change needed.
  1. Class names are clear adjective-style: Equatable, Comparable, Hashable, Sizable, Validatable.
  2. Has* mixins follow a consistent naming pattern for property injection.
  3. All docstrings follow the ==== separator convention.
  4. Module folder names clearly indicate their purpose.

3. Design (OOP)

Clean design.
  1. Single-responsibility: each mixin adds exactly one capability. No mixin exceeds 40 lines.
  2. Key-based identity: Equatable → Comparable → Hashable chain is clean. Single abstract key drives all operators.
  3. Orthogonal hierarchies: equality chain and size chain are independent. Dictable extends Sizable, not Equatable — implements its own __eq__.
  4. Validatable split: immutable vs mutable validity is a clean separation. Default False in mutable variant is a safe default.
  5. Has* mixins: correctly split into identity mixins (with Comparable/Hashable) and standalone property mixins (without).

4. Performance

No change needed.
  1. Comparable implements all 4 operators explicitly instead of using @total_ordering. Each operator is a single key comparison — no overhead.
  2. All mixins are lightweight: no allocation, no caching, O(1) property access.
  3. Dictable delegates to dict internals — no performance penalty over plain dict.

5. Typing

Minor issues found.
  1. dictable/main.py:18 — data: dict[K, V] = None should be dict[K, V] | None = None.
  2. dictable/main.py:50 — default: V = None should be V | None = None.
  3. comparable/main.py:26,34,42,50 — All comparison methods accept other: object but access other.key without narrowing. Technically correct for internal use but unsafe for public API.
  4. All other type annotations are correct and complete.

6. Documentation

No change needed.
  1. All classes and methods have docstrings following the ==== separator convention.
  2. Equatable docstring explicitly documents why __ne__ is omitted.
  3. Hashable docstring documents the a == b implies hash(a) == hash(b) invariant.

7. Modularity

No change needed.
  1. Each mixin in its own folder with standard file layout (main.py, _factory.py, _tester.py, __init__.py).
  2. Package __init__.py exports 10 most-used classes. 3 less-used classes require direct import.
  3. No circular dependencies between any modules.
  4. has/ sub-package is self-contained with its own __init__.py and CLAUDE files.

8. Architecture

Minor inconsistencies.
  1. comparable/__init__.py:1 — Imports Comparable from _factory.py instead of main.py. 
    # current
from f_core.mixins.comparable._factory import Factory, Comparable

# convention
from f_core.mixins.comparable.main import Comparable
from f_core.mixins.comparable._factory import Factory
  2. dictable/__init__.py:1 — Same pattern: imports Dictable from _factory.py.
  3. equatable, hashable, validatable, validatable_mutable correctly import from main.py.
  4. All imports work correctly — this is a convention consistency issue, not a bug.

9. Usage & Usefulness

  1. How to use: inherit from one or more mixins, implement abstract methods (key for Equatable/Comparable, __len__ for Sizable).
  2. When useful: every domain class in the framework uses these mixins. They are the foundation of the entire OOP design.
  3. Overengineered? No. Each mixin is 10-40 lines. The hierarchy is shallow and each class has a clear single purpose.
  4. Design strength: single abstract key property drives equality, ordering, and hashing — implement once, get all operators.
  5. Composition works well: HasRowCol(Comparable, Hashable), ProcessBase(HasName), Solutions(Dictable, Validatable) are all clean compositions.
