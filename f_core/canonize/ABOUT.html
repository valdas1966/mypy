What is Canonicalization?

Turning a tangle of nested objects into one clean, comparable primitive — the same identity, every time.

f_core/canonize · canonize(v)

The Idea

🪆 A matryoshka doll. You keep opening it — doll inside doll inside doll — until you reach the small solid core that can't open any further. canonize does exactly this with identity: it keeps descending .key until it hits a plain primitive — a tuple, an int, a string — the irreducible core of "who is this?".

In the framework, identity is layered. A StateCell is identified by a CellMap, which is identified by its (row, col). That nesting is great for code — but useless for a test or a chart, which need a flat literal like (0, 0) they can print and compare with ==. Canonicalization is the bridge between the two worlds.

✗ Without canonize

Every State subclass re-hand-rolls its own unwrap:

class StateCell:
    def event_key(self):
        return self.rc          # (row,col)

class StateRC:                # next subclass…
    def event_key(self):        # …copies the
        return (self.rc, self.b)  # same idea again

✓ With canonize

One function descends .key for any keyed object:

# StateCell, StateRC, graph states…
# all just work — zero per-class code
canonize(state_cell)  → (0, 0)
canonize(state_rc)    → ((0, 0), 5)
canonize(graph_state) → 'A'
canonize(42)          → 42

Three rules — that's the whole function

  1. Keyed object (HasKey / HasRowCol) → descend into its .key and canonize that.
  2. Container (tuple / list) → canonize each element, keep the container type.
  3. Anything else (int, str, None, …) → it's already primitive, return it unchanged.

How It Descends

One call unwinds the whole identity stack, layer by layer, until only primitives remain:

canonize(StateCell) // HasKey → descend .key └─ canonize(CellMap) // HasRowCol → descend .key └─ canonize((0, 0)) // tuple → element-wise ├─ canonize(0) → 0 // primitive └─ canonize(0) → 0 // primitive ▼ (0, 0) // flat, comparable, printable

This Module's Contribution

f_core/canonize/u_canonize.py — the entire implementation 
def canonize(v: object) -> object:
    # Keyed object → descend into its key
    if isinstance(v, (HasKey, HasRowCol)):
        return canonize(v.key)
    # Container → canonize element-wise, preserving type
    if isinstance(v, (tuple, list)):
        return type(v)(canonize(x) for x in v)
    # Primitive / None → unchanged
    return v

Why a free function, not a method or mixin?

🍃
Leaf types aren't yours
It must handle tuple, int, None — you can't add (0,0).canonize() to a built-in. A free function can.
👯
Two sibling mixins
HasRowCol does not inherit HasKey. A single method couldn't cover both; the function's isinstance covers both in one line.
🧬
No per-type variation
The descent rule is universal — zero overrides. Variation is by shape, which one function expresses better than scattered methods.

Why it lives in f_core — the layer-floor rule

A utility must sit at or below the lowest layer of the types it touches. canonize descends mixins from f_core, and must flatten a CellMap (an f_ds type) and States (f_hs types). Only f_core sits below all of them — so every consumer reaches it downward, never creating a backward import.

f_core # HasKey, HasRowCol, canonize live here — the floor ▲ f_ds # CellMap (HasRowCol) — canonize flattens it ▲ f_hs # StateCell, StateRC — canonize flattens them too # Put canonize in f_hs instead → f_ds would import UP to reach it. Illegal.

Golden references are hand-authored Python literals compared with ==. A live StateCell object can't appear in a literal — canonize turns it into the (0, 0) the golden expects. Used by every algo's _tester_recording.py via u_event_normalize.normalize.

Fun Facts & History

📐
"Canonical" comes from canon law

A canon was the official, authoritative list. In math & CS a "canonical form" is the one official representative of a whole equivalence class — pick it, and equal things look equal.

♻️
Idempotence is the safety net

canonize(canonize(v)) == canonize(v). Once you've reached the primitive core, running it again changes nothing — so it's always safe to canonize "just in case."

🔑
Unicode does this billions of times a day

Unicode normalization (NFC/NFD) canonicalizes accented characters so "é" typed two different ways compares equal — the exact same problem, on text instead of search states.

✂️
It replaced a method that bred copies

Before, each State carried an event_key(). The moment a second subclass arrived, it copied the unwrap. canonize killed the method and the copies in one move — written once.
