1. What is an HTTP Client?

An HTTP client is any program that sends requests to a server and receives responses. Your browser, curl, a Python script — all are HTTP clients.

2. Where Does the Name Come From?

Etymology of "Client"
Latin: cliēns → one who leans on another → dependent, follower
In ancient Rome, a cliēns was a citizen who depended on a patrōnus (patron) for protection and services. The patron held the resources; the client made requests.

In 1960s computing, Xerox PARC and MIT adopted the term for the client-server model: the program that requests a service (client) vs. the program that provides it (server). The metaphor is perfect — the client depends on the server, just as the Roman citizen depended on his patron.

Timeline: Client-Server in Computing

1960s
Client-server model coined
Emerged from time-sharing systems at MIT and Xerox PARC. Terminals (clients) requested computation from mainframes (servers).
1991
First HTTP client: WorldWideWeb browser
Tim Berners-Lee wrote the first web browser at CERN. It was both client and editor — you could browse and edit pages.
1996
HTTP/1.0 formalized
RFC 1945 defined GET, POST, HEAD. Clients now spoke a standard protocol with status codes and headers.
1998
XMLHttpRequest (XHR) born
Microsoft created XHR for Outlook Web Access. Later adopted by all browsers, enabling AJAX — clients that update without full page reloads.
2011
Python requests library released
Kenneth Reitz released "HTTP for Humans." Became the most downloaded Python package. Our Client class wraps it.
2015
HTTP/2 — multiplexed clients
Single TCP connection, multiple parallel requests. Clients got dramatically faster without code changes.

Usage

# One call, one object, no exceptions
from f_http.client import Client

response = Client.get_json(url='https://api.example.com/data')

if response:
    print(response.data)           # parsed JSON dict
    print(response.elapsed)        # 0.23 seconds
else:
    print(response.status.code)    # 404
    print(response.exception)      # error message or None

4. Fun & Curious Facts

🌍
The First HTTP Client Was Also an Editor
Tim Berners-Lee's "WorldWideWeb" (1990) wasn't just a browser — it could edit pages too. The web was designed to be read-write from day one. We lost that vision and got read-only browsers. It took until Web 2.0 (2004+) to get writing back via JavaScript and APIs.
📦
requests: 300 Million Downloads/Month
Python's requests library (which our Client wraps) is downloaded ~300 million times per month. Kenneth Reitz wrote it in 2011 because urllib2 was "a work of insanity." Its tagline: "HTTP for Humans." The first commit was 131 lines.
⚡
curl: The Swiss Army Knife
curl (1998) is the most widely deployed HTTP client in history — installed on virtually every Linux, macOS, and Windows 10+ machine. Daniel Stenberg has maintained it for 26+ years as a solo project. It supports 28 protocols, but started as a tool to fetch currency exchange rates.
🤖
Most HTTP Clients Aren't Human
Over 60% of all HTTP traffic comes from bots, crawlers, and automated clients. Googlebot alone makes billions of requests daily. Your API wrapper script is in good company — most of the web is machines talking to machines.
🔥
The Timeout That Broke the Internet
In 2017, an Amazon S3 outage cascaded because HTTP clients across the internet had no timeouts or excessively long retry loops. Services waited forever for S3 to respond, exhausting their own connection pools. Half the internet went down because clients didn't know when to give up. (Our Client defaults to 30s.)
📜
User-Agent: A History in One Header
Every HTTP client sends a User-Agent header identifying itself. Modern Chrome sends: "Mozilla/5.0 ... Chrome/120 ... Safari/537.36" — pretending to be Mozilla, Chrome, AND Safari simultaneously. This absurd string exists because servers in the 1990s served different content based on the browser name, so every new browser had to lie about being the old ones too.
🔄
Keep-Alive: The Optimization Nobody Sees
HTTP/1.0 opened a new TCP connection for every single request. Loading a page with 50 images meant 50 separate TCP handshakes. HTTP/1.1 (1997) added Keep-Alive to reuse connections. HTTP/2 (2015) went further with multiplexing — many requests, one connection. HTTP/3 (2022) switched to QUIC, eliminating TCP entirely.
