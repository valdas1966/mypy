1. What Are Algorithm Stats?

Every algorithm does work. Stats are the receipt.

Algorithm stats are quantitative measurements captured during an algorithm's execution that answer three questions:

  1. How long? — Wall-clock time consumed
  2. How much? — Resources spent (nodes, memory, operations)
  3. How good? — Quality of the result (optimality, accuracy)

Without stats, two algorithms that return the same answer look identical. With stats, one might be 100x faster or explore 1/10th the nodes.

3. The Stats Pipeline

Stats flow through the algorithm lifecycle:

Start
t = 0
→
Discover
discovered++
→
Relax
relaxed++
→
Explore
explored++
→
Done
elapsed = t

Each counter increments at a specific moment. At the end, elapsed captures total wall-clock time.

4. Our Model

Two-layer design — generic base, domain-specific extension:

StatsAlgo                 # Layer 1: Any algorithm
    elapsed                  time consumed

    +-- StatsSearch          # Layer 2: Search algorithms
           discovered           states added to frontier
           relaxed              states updated with better cost
           explored             states fully expanded

Design Decisions

  • Mutable: counters increment during the run via setters
  • Flat: no SPP/OMSPP stats subclasses — StatsSearch serves both; solutions handle aggregation
  • Minimal: only 4 fields — just enough to compare any two search algorithms

5. Stats Catalog

Every search algorithm stats object should carry these metrics:

#StatTypeUnitMeaning
1 elapsed int seconds Wall-clock time from start to finish
2 discovered int count States added to the open list (frontier)
3 relaxed int count States updated with a shorter path (re-prioritized)
4 explored int count States popped from frontier and fully expanded

Invariants

explored  ≤  discovered      # can't explore what wasn't found
relaxed   ≤  discovered      # relaxation is a subset of discovery
explored  +  frontier.size  =  discovered - duplicates

Derived Metrics (computed in Solution, not in Stats)

MetricFormulaRangeMeaning
quality_h h(start) / g(goal) [0, 1] 1.0 = perfect heuristic; lower = less informed
branching_eff discovered / explored ≥ 1 Average successors per expansion
relaxation_rate relaxed / discovered [0, 1] Fraction of states that needed re-ordering

6. Key Relations

How stats connect to algorithmic concepts:

   discovered          All states the algorithm has seen.
       |                The "frontier" is the set of discovered
       |                but not yet explored states.
       v
   relaxed             A discovered state got a better path.
       |                Its priority in the queue changes.
       |                Good heuristics → fewer relaxations.
       v
   explored            A state is popped, expanded, done.
       |                This is the main cost driver.
       |                A* explores fewer than Dijkstra.
       v
   elapsed             The bottom line. Everything above
                        contributes to this number.

What Each Stat Tells You

If this is high...It means...
exploredAlgorithm searches too broadly; heuristic is weak
relaxedMany paths compete; graph has lots of similar-cost routes
discoveredLarge frontier; high branching factor
elapsedTime to optimize (better heuristic, pruning, caching)

7. Fun Facts

The First "Stats"

When Edsger Dijkstra published his shortest-path algorithm in 1959, the paper was 3 pages long and contained zero performance measurements. He just proved it was correct and moved on.

The A* Asterisk

The * in A* stands for "optimal among all algorithms using the same heuristic." Hart, Nilsson & Raphael (1968) proved that no other algorithm can explore fewer nodes than A* with a consistent heuristic — making explored the perfect stat to compare against.

10x vs 10,000x

On a 1000×1000 grid, Dijkstra explores roughly 500,000 nodes. A* with Manhattan distance? About 5,000. Same answer, 100x less work. That ratio is the stats.

The "Explored" Paradox

An algorithm that explores fewer states isn't always faster. If each exploration involves an expensive heuristic calculation, elapsed can go up while explored goes down. Stats never lie — but you need all of them to see the truth.

Knuth's Estimate

In 1975, Donald Knuth proposed estimating a search tree's total size by random sampling — essentially predicting explored before the search even runs. The technique is still used in SAT solvers today.

NASA Uses These Exact Stats

Mars rover path planning uses A* on terrain grids and logs discovered, explored, and elapsed — because when you're 225 million km from the nearest debugger, stats are all you have.

8. Historical Timeline

1959
Dijkstra's Algorithm — First systematic shortest-path algorithm. No stats recorded; correctness was the only metric that mattered.
1968
A* Published — Hart, Nilsson & Raphael introduce heuristic search. The paper compares algorithms by number of nodes expanded — the birth of explored as a standard metric.
1975
Knuth's Tree Estimation — Predicting search tree size via random walks. Early recognition that stats can be estimated, not just measured.
1985
IDA* (Korf) — Iterative-deepening A*. Introduced the trade-off: more explored nodes, but near-zero memory. Stats now needed to capture space alongside time.
1992
DIMACS Challenges — Standardized benchmarks for graph algorithms. For the first time, researchers compared stats across implementations on identical inputs.
2004
Moving AI Benchmarks — Nathan Sturtevant's grid-based pathfinding benchmarks become standard. Every paper now reports explored and elapsed on the same maps.
2026
This Framework — StatsAlgo and StatsSearch capture the essential metrics in a clean, extensible two-layer model. Four fields. Zero waste.
