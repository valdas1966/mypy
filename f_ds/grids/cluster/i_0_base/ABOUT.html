What is a Grid Cluster?

A named blob of cells that belong together on a grid — the abstract contract every shape must fill.

The Idea

🏘️ A grid cluster is a neighborhood. Pick a spot on a city map, draw a boundary around the blocks that belong together, give it a name. The boundary's shape can vary — a circle, a diamond, a zip-code blob — but every neighborhood is the same kind of thing: a labeled set of cells. ClusterGrid is the word "neighborhood"; the shapes are the actual zones.

ClusterGrid is abstract. It never decides which cells are in. It only guarantees what any cluster-of-cells can do, then hands the shape question to subclasses.

🧩

Without the base

Every shape re-implements len, membership, iteration, naming, the "which grid did I come from?" bookkeeping, and the representative slot. Five shapes → five copies of the same plumbing, drifting apart.

✅

With ClusterGrid

The plumbing lives once in the base. A new shape writes only its _build + how it fills _cells. len/in/iter/bool, name, map, cells all come for free.

Five properties of a grid cluster

  1. Its members are cells. Specifically CellMaps — valid (non-wall) squares of a GridMap.
  2. It is light. It stores the grid's name (map: str), never the grid object. Pickle small; outlive the in-memory grid.
  3. It is named. Identity / label via the HasName mixin.
  4. It may have a representative. One distinguished cell (a center / seed). Optional — default None.
  5. It is shape-agnostic. The base says nothing about which cells. Subclasses answer that with to_iterable().

Where It Sits

ClusterGrid is the root of the cluster hierarchy. It composes two mixins — collection behaviour and a name — then leaves the shape to its subclasses.

This Layer's Two Classes

This package ships an abstract base and one concrete shape. They are not rivals — one defines the contract, the other fills it. Knowing which is which is the whole point.

ClusterGrid abstract · i_0_base

The contract. "A named, light set of cells on a grid, with an optional representative." Cannot be instantiated. Knows the what, not the which.

Use it as a type (cluster: ClusterGrid) and as the thing you subclass to invent a new shape.

ClusterDiamond concrete · i_1_diamond

One filling of the contract. "All cells within steps Manhattan moves of a center." BFS from a seed; walls dent it. Can be instantiated and used.

It answers the which the base left open — see its own ABOUT.html.

What This Module Adds

Beyond the two mixins it composes, ClusterGrid contributes two grid-specific ideas:

🪶 Grid provenance, not the grid

It snapshots grid.name into map: str and drops the grid. A cluster can be pickled, cached, or shipped without dragging a whole GridMap along.

🔲 A cells-flavored view

It exposes the members as cells (a grid-named alias for members) and implements to_iterable() over a list[CellMap] that subclasses fill.

class ClusterGrid(Collectionable[CellMap], HasName, ABC):
    """Abstract: a set of valid CellMaps on a GridMap.
       Holds only the grid's NAME, not the grid object."""

    def __init__(self, grid: GridMap) -> None:
        HasName.__init__(self, name=type(self).__name__)
        self._map: str = grid.name        # provenance, not the grid
        self._cells: list[CellMap] = []   # subclass fills via _build()

    @property
    def map(self) -> str:  # the grid's name — the only identity kept
        return self._map

    def to_iterable(self) -> list[CellMap]:  # drives len/in/iter/bool
        return self._cells

Notice what's absent: no _build, no geometry, no steps. Those belong to a shape. The base stays empty on purpose — that emptiness is the contract.

The Shape Catalog

One shape ships today; the base was built so the rest cost almost nothing. Each is a subclass that only writes its _build.

🔷

ClusterDiamond shipped

Manhattan ball: |Δr|+|Δc| ≤ steps. 4-connected BFS. 2s²+2s+1 cells (wall-free).

🟦

ClusterSquare idea

Chebyshev ball: max(|Δr|,|Δc|) ≤ steps. 8-connected BFS. A flat-sided sibling.

⚪

ClusterDisk idea

Euclidean ball: Δr²+Δc² ≤ r². The roundest neighborhood the grid can approximate.

All three would share everything in this base — the only difference is the predicate inside _build. That is the payoff of an abstract base done right.

Fun Facts & History

🧬 "Cluster" is older than computers

The word traces to Old English clyster — a bunch of grapes. Statistical cluster analysis dates to the 1930s (Driver & Kroeber, anthropology), long before k-means (1957).

🪶 The flyweight trick

Storing the grid's name instead of the grid is the Flyweight pattern: keep a cheap handle to heavy shared state. Thousands of clusters, one grid in memory.

🚫 An abstract class is a promise

Python's ABC + @abstractmethod makes ClusterGrid() raise TypeError at construction. The compiler-enforced "you must pick a shape."

🧱 Walls make clusters topological

Because membership is decided by BFS over real grid neighbors, a cluster is a connected component, not a paint-by-coordinates region. Geometry proposes; the grid disposes.
