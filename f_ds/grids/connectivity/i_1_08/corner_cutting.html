Corner Cutting  ·  Connectivity_8

Why a diagonal step can be illegal even when its destination cell is perfectly free — and the exact rule is_legal_move enforces.

1 · Cardinals enter through an edge; diagonals pivot on a corner

Cardinal — slides in through an open side
Diagonal — squeezes past a single point

A cardinal step crosses a whole shared edge into the target — nothing else is in the way, so only the target's own freeness matters. Cardinals are therefore always legal.

A diagonal step instead rounds a single corner point shared by four cells. An agent with real width brushes whatever sits beside that corner — so the diagonal needs a second look.

2 · What a diagonal brushes — the two flank cells

The diagonal A → B clips the corner shared by four cells. Two of them — the flanks — sit on either side of that corner: the cell in A's row, B's column and the cell in B's row, A's column.

As the agent rounds the corner it grazes both flanks. So the move is legal only when both flank cells are free:

flank₁ = is_free(a.row, b.col)
flank₂ = is_free(b.row, a.col)

3 · The four corner cases

flank (free) flank (wall) A / B (free)

A diagonal is legal in exactly one of the four flank configurations — the one where both are open.

✓ LEGALboth flanks free
✗ ILLEGALclips the wall corner
✗ ILLEGALclips the wall corner
✗ ILLEGALwalls touch — no gap
Case 4 is the decisive one. A and B are free, yet the two walls meet at exactly the corner point the diagonal passes through. There is no real gap to slip through — they touch. Allowing this move would let the path travel through solid wall.

4 · The rule, in code

def is_legal_move(self, a, b, is_free) -> bool:
    d_row = b.row - a.row
    d_col = b.col - a.col
    is_cardinal = d_row == 0 or d_col == 0
    if is_cardinal:
        return True                # edge entry — always legal
    flank_row_free = is_free(a.row, b.col)  # A's row, B's col
    flank_col_free = is_free(b.row, a.col)  # B's row, A's col
    return flank_row_free and flank_col_free

Note what is not here: the destination b itself. The grid already verified b is free when it enumerated the neighbor; this policy adds only the extra corner constraint that a diagonal — and only a diagonal — must satisfy.

5 · Why “both”, not “either”?

There are three corner-cutting conventions. Connectivity_8 takes the strict one:

ConventionDiagonal allowed when… Case 4?
Crossing always (corner never checked) passes ✗
Lenient either flank free (≥ 1 open) passes ✗
Strict  (ours) both flanks free blocked ✓
Strict is the benchmark standard. MovingAI, GPPC and JPS publish optimal path costs under strict no-corner-cutting. Matching it means our solver's costs are directly comparable to those reference optima — and it keeps the octile heuristic an honest lower bound, since forbidding corner-cuts only ever makes the real path longer, never shorter.
